STANDING RULE (2026-06-10): Wolf's bootstrap + RBAC model follows the Wazuh pattern. Captured here so it survives across sessions + is treated as a load-bearing design constraint.

## The bootstrap flow

1. **Install** — `.deb` postinst (or `bootstrap_superuser.sh` wrapper script) creates the Superuser. Username FIXED: `Wolf`. Password: 32+ char random, AUTOGENERATED at install time, **printed once** to the operator's stdout (then never shown again — operator stores it themselves, e.g., in 1Password).
2. **First login** — operator logs in as `Wolf`. Initial state: zero organizations, zero other users, zero Wazuh component mappings.
3. **Wazuh component mapping** — Superuser is the ONLY user who can configure + map Wazuh components (indexer URLs/creds, manager URLs/creds, dashboard URL). Supports both deployment shapes: single-host AND distributed (master + worker cluster). UI in the Superuser-only Settings section.
4. **Organization creation** — Superuser creates organizations. Org must exist BEFORE any non-superuser user can be created.
5. **First-user-per-org assignment** — Superuser creates the first User per organization + assigns role = `Admin` within that org.
6. **Org-admin onboarding chain** — Each org's Admin creates further users within their org + assigns per-org roles (Engineer / Analyst / future roles).

## Roles (per-organization, on UserOrganization not on User)

| Role | Capabilities |
|---|---|
| **Superuser** | Create/edit/delete organizations + users + assign roles. Configure + map Wazuh components (single + distributed). **Cannot** access any organization's data unless the **organization's Admin explicitly grants membership** (org-consent gate, 2026-06-10). Superuser cannot self-grant. There is exactly ONE Superuser per Wolf install (the bootstrap `Wolf` user). |
| **Admin** (per-org) | Full access to their organization: configure org settings, manage users within the org, assign roles within the org, all data access. Also: grants or revokes Superuser membership in the org (the consent gate). Propose + approve actions. |
| **Engineer** (per-org) | Configure + customize + fine-tune Wolf for the org: RAG corpus management, prompt customization, wolf-pack deployment, model selection, embedding provider. Plus chat + read all data, propose + approve actions. **Does NOT configure Wazuh component mapping** (Superuser-only) or manage users. |
| **Responder** (per-org; renamed from Approver 2026-06-10) | Incident-response role. Chat, read all data, view audit log, propose + approve actions, AND **execute actions directly** (the only role with the direct-execute shortcut — for time-critical containment). Audit-after, not approve-before. |
| **Analyst** (per-org) | Chat + read-only access to the org's data + own conversation history + **propose actions** (someone else must approve). Cannot approve, cannot execute, cannot configure. |
| (future) | Extensible — Auditor (read-only audit log) / Read-only (Analyst minus propose) / Custom (operator-defined). |

## Why a single fixed-username Superuser

- **Wazuh-pattern familiarity** — operators coming from Wazuh recognise the model (Wazuh ships a fixed admin user too).
- **Audit clarity** — there is exactly ONE Superuser identity; security events attributable to `Wolf` are unambiguous.
- **Reduced attack surface** — fewer privileged identities to defend.
- **Operationally simple** — no "first run wizard" complexity. The bootstrap process knows exactly which user to create.

## What the Superuser CANNOT do

- Access organization data without explicit UserOrganization membership grant (superuser is an ADMIN permission, not a data-access bypass).
- Be created at runtime (the install-time bootstrap is the only way).
- Be deleted or renamed (the install would be unusable without a Superuser).
- Be subjected to MFA bypass or password recovery via email — recovery goes through the `bootstrap_superuser.sh` CLI wrapper instead (operator-on-host action, audit-logged).

**Why:** [[shell-wrapper-required-pattern]] for the recovery path; [[integrity-across-the-stack]] for the no-bypass rule.

## How to apply

- Any feature touching auth / RBAC / org-creation / Wazuh-mapping must respect this hierarchy.
- ADR 0018 (proposed; bootstrap + RBAC + login UX + Wazuh-mapping) is the canonical written form of this rule.
- If a feature requires a user to do something this hierarchy doesn't permit, push back: either expand the role model (and capture the change here) OR find a different path. Never silently grant permissions outside the model.
